● Designed for operations

From signal to action, without switching tools.

Incidents don't wait while you juggle terminals, dashboards, and YAML files. srelens keeps the whole investigation in one connected workspace — from signal to action.

  1. 01 — discover

    Discover

    Operate

● Feature surface

Everything an operator reaches for.

resources

40+ resource kinds

Workloads, config, storage, networking, RBAC, admission, autoscaling — filterable by namespace, remembered per cluster.

live data

Streaming watches

Tables update in place as the cluster changes, backed by shared kube-rs watch streams — not polling.

speed

Command palette

Hit ⌘K to jump to any resource in any cluster — global search with deep links into detail views.

ecosystem

Helm & CRDs

Inspect Helm releases and browse custom resources grouped by API group — CRDs are first-class citizens.

metrics

Cluster usage

CPU and memory sparklines for nodes and pods via metrics-server — health at a glance.

node ops

Safe cluster actions

Scale, rollout-restart, evict, cordon, drain — every destructive action is flagged and confirmation-gated.

contexts

Multi-kubeconfig

Add kubeconfig files or paste one in. Context profiles, custom ordering, and a hotbar for the clusters you live in.

tabs

Browser-style tabs

Each tab is a cluster + resource view. Close others, close to the right — investigation state that survives.

privacy

Local-first

Talks directly to your API servers with your kubeconfig credentials. No cloud service in between.

● Why not Electron?

The Lens workflow, without the Lens weight.

srelens is inspired by Lens and Freelens — it rebuilds what made them great and removes the machinery that made them heavy.

srelens Electron-era Kubernetes IDEs
Rendering OS system WebView via Tauri v2 Bundled Chromium shipped in every install
Core runtime Pure Rust — no Node.js in the core app Node.js main process
Cluster access kube-rs, direct to the API server Bundled kubectl + helm binaries behind a proxy layer
AI agents Built-in MCP server, every capability exposed —
UI stack React 19, rebuilt clean-room Legacy React 17 + MUI lineage

● FAQ

Questions operators ask.

What is srelens?

srelens is a Kubernetes desktop workspace — a native GUI app for browsing, inspecting, and operating Kubernetes clusters. It reads the contexts in your local kubeconfig and gives you resource browsing across 40+ kinds, live watches, log streaming, in-pod terminals, port forwarding, Helm release views, and a schema-aware YAML editor, all in one window. It's built on Tauri v2 with a pure-Rust core.

How is srelens different from Lens or Freelens?

Lens and Freelens are Electron apps: they ship a bundled Chromium browser and a Node.js main process, and reach clusters through bundled kubectl and helm binaries plus a proxy layer. srelens rebuilds the same operator workflow on Tauri v2 — it renders in your OS system WebView and its core is pure Rust, talking to the API server directly through kube-rs. That means a much smaller binary, lower memory use, and no Node.js in the core app. It also adds something the Lens family doesn't have: a built-in MCP server for AI agents.

What platforms does srelens run on?

macOS, Windows, and Linux desktops via Tauri v2. Android groundwork exists in the codebase, but desktop is the focus.

Does srelens replace kubectl?

No — it complements it. srelens talks to the Kubernetes API directly through kube-rs rather than shelling out to kubectl, so it doesn't need kubectl installed. But it's a workspace for investigation and day-to-day operations, not a replacement for the scripting and automation you do in the terminal or CI.

How does the AI (MCP) integration work?

Every backend operation — listing pods, streaming logs, scaling a deployment, draining a node — is registered as a capability in one shared registry, and each capability is automatically exposed as a Model Context Protocol tool by the built-in MCP server. Agents connect over stdio (Claude Desktop, IDEs) or loopback HTTP and call the same operations the GUI uses. Destructive tools are consent-gated: nothing gets deleted or drained without explicit approval.

Where do my cluster credentials go?

Nowhere. srelens runs entirely on your machine and connects to clusters directly using the credentials in your local kubeconfig files. There's no intermediary cloud service between the app and your API servers.

Can I download srelens today?

srelens is in pre-release development (v0.1.x). Installers aren't published yet, but you can build and run it from source with pnpm and the Rust toolchain. Follow the repo on GitHub for release announcements.
