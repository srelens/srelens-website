● The reliability loop

From signal to action,
without switching tools.

Incidents don't wait while you juggle terminals, dashboards, and YAML files. srelens keeps the whole investigation in one connected workspace.

  1. 01 — observe

    Observe

    Move through workloads, networking, storage, and access control without changing tools. Every context, one hotbar away. Live watches stream state as it changes.

  2. 02 — diagnose

    Diagnose

    Inspect health, relationships, events, and manifests side by side. Follow logs, exec into containers, and forward ports — all docked next to the evidence.

  3. 03 — act

    Act

    Scale, restart, cordon, drain, or apply an edit right where you found the problem. Destructive actions always ask first — your error budget stays yours.

Every capability is an MCP tool

One registry drives the GUI and a built-in MCP server — backend capabilities are also available as MCP tools. Claude Desktop and MCP-capable IDEs can connect over stdio or loopback HTTP. Mutating tools require an explicit _confirm: true argument before they run.

● Why not Electron?

A familiar workflow, rebuilt in Rust.

srelens follows a familiar Kubernetes desktop workflow while using Tauri v2 and a pure-Rust core. It is independently developed and is not affiliated with Mirantis Lens or the Freelens project.

● FAQ

Questions operators ask.

What is srelens?

srelens is a Kubernetes desktop workspace — a native GUI app for browsing, inspecting, and operating Kubernetes clusters. It reads the contexts in your local kubeconfig and gives you resource browsing across 40+ kinds, live watches, log streaming, in-pod terminals, port forwarding, Helm release views, and a schema-aware YAML editor, all in one window. It's built on Tauri v2 with a pure-Rust core.

How is srelens different from Lens or Freelens?

Lens and Freelens are Electron apps: they ship a bundled Chromium browser and a Node.js main process, and reach clusters through bundled kubectl and helm binaries plus a proxy layer. srelens rebuilds the same operator workflow on Tauri v2 — it renders in your OS system WebView and its core is pure Rust, talking to the API server directly through kube-rs. This avoids bundling Chromium and keeps Node.js out of the core application runtime. It also adds something the Lens family doesn't have: a built-in MCP server for AI agents.

What platforms does srelens run on?

macOS, Windows, and Linux desktops via Tauri v2. Android groundwork exists in the codebase, but desktop is the focus.

Does srelens replace kubectl?

No — it complements it. srelens talks to the Kubernetes API directly through kube-rs rather than shelling out to kubectl, so it doesn't need kubectl installed. But it's a workspace for investigation and day-to-day operations, not a replacement for the scripting and automation you do in the terminal or CI.

How does the AI (MCP) integration work?

Every backend operation — listing pods, streaming logs, scaling a deployment, draining a node — is registered as a capability in one shared registry, and each capability is automatically exposed as a Model Context Protocol tool by the built-in MCP server. Agents connect over stdio (Claude Desktop, IDEs) or loopback HTTP and call the same operations the GUI uses. Destructive tools are consent-gated: nothing gets deleted or drained without explicit approval.

Where do my cluster credentials go?

Nowhere. srelens runs entirely on your machine and connects to clusters directly using the credentials in your local kubeconfig files. There's no intermediary cloud service between the app and your API servers.

Can I download srelens today?

Yes — pre-release installers for macOS (Apple Silicon and Intel), Windows, and Linux (AppImage, deb, rpm) are on the download section and GitHub Releases. You can also build from source with pnpm and the Rust toolchain.
